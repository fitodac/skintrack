# Local development

## Recommended flow

The current project favors a single Docker-first entrypoint:

```bash
npm run local:up
```

This gives you:

- local Supabase through the CLI
- generated local app credentials in `.env.local`
- regenerated Supabase types
- the Next.js app in the `skintrack-app` container

## Main scripts

- `npm run local:setup`: start local Supabase, write `.env.local`, regenerate types
- `npm run local:up`: run `local:setup` and start `skintrack-app`
- `npm run local:down`: stop `skintrack-app` and stop local Supabase
- `npm run db:start`: start local Supabase only
- `npm run db:status`: inspect local Supabase status
- `npm run db:reset`: rebuild local DB from migrations
- `npm run db:stop`: stop local Supabase only
- `npm run db:sync:prod-to-local`: reset local DB and copy production data into it

## Containers you should see

After `npm run local:up`, Docker Desktop should show:

- `skintrack-app`
- Supabase local containers created by the CLI

Common Supabase container names include:

- `supabase_db_skintrack`
- `supabase_kong_skintrack`
- `supabase_auth_skintrack`
- `supabase_rest_skintrack`
- `supabase_studio_skintrack`

## Generated env model

`.env.local.example` is intentionally split in two categories:

- values you set yourself, such as `NEXT_PUBLIC_APP_URL` and `SUPABASE_PRODUCTION_DB_URL`
- values generated by local setup, such as `NEXT_PUBLIC_SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`

The generated `SUPABASE_SERVER_URL` is the server-side URL used by the app container. It points to `host.docker.internal` so server code can reach local Supabase even though the app itself runs in Docker.

## Production-to-local sync

Use this when local development must reflect real data:

```bash
npm run db:sync:prod-to-local
```

What it does:

1. starts local Supabase if needed
2. resets the local database from repo migrations
3. lists `public` base tables from production
4. dumps `public.*`, `auth.users`, and `auth.identities`
5. restores the dump into local Supabase

What it does not copy:

- active sessions
- refresh tokens

## Connection string rule

For `SUPABASE_PRODUCTION_DB_URL`, a direct Supabase host must use the `postgres` user:

```env
SUPABASE_PRODUCTION_DB_URL=postgresql://postgres:your-db-password@db.your-project-ref.supabase.co:5432/postgres
```

Do not use a pooler-style username such as `postgres.your-project-ref` with that direct host.

## When to restart the stack

Restart the full local stack after changes to:

- `supabase/config.toml`
- auth provider settings used by local Supabase
- generated env values that affect the app container

Recommended restart:

```bash
npm run local:down
npm run local:up
```

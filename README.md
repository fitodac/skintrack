# SkinTrack

SkinTrack is a Next.js app for cosmetology practices.

It includes:

- private staff authentication
- patient records
- clinical session drafts with autosave
- superadmin user management

## What You Need Installed

Install these tools before you do anything else:

- Node.js 20+
- npm
- Docker Desktop
- PostgreSQL client tools: `psql` and `pg_dump`

Why Docker is involved:

- Supabase local runs in Docker containers managed by Supabase CLI
- the Next.js app runs in its own Docker container from this repo

After the local setup works, you should see both the app container and the Supabase containers in Docker Desktop.

## Installation

Run these commands in order:

```bash
npm install
cp .env.local.example .env.local
```

Now open `.env.local` and fill only the values that are really project-specific.

At minimum, make sure these are correct:

```env
NEXT_PUBLIC_APP_URL=http://localhost:3000
SUPABASE_PRODUCTION_DB_URL=postgresql://postgres:your-db-password@db.your-project-ref.supabase.co:5432/postgres
```

If you want Google login in local too, also fill:

```env
SUPABASE_AUTH_EXTERNAL_GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
SUPABASE_AUTH_EXTERNAL_GOOGLE_SECRET=your-google-client-secret
```

If you do not fill the Google variables, the project can still run locally with email/password.

## .env.local Structure

Copy [`.env.local.example`](/Users/fitodac/.codex/worktrees/4e51/skintrack/.env.local.example) to `.env.local`.

This is the exact structure:

```env
# App URL
NEXT_PUBLIC_APP_URL=http://localhost:3000

# These values are written automatically by `npm run local:setup`.
# You can leave them as-is before the first run.
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321
NEXT_PUBLIC_SUPABASE_ANON_KEY=run-npm-run-local-setup
SUPABASE_SERVICE_ROLE_KEY=run-npm-run-local-setup
SUPABASE_SERVER_URL=http://host.docker.internal:54321

# Production-to-local database sync.
# Use the Supabase direct connection string.
# If the host is `db.<project-ref>.supabase.co`, the username must be `postgres`.
SUPABASE_PRODUCTION_DB_URL=postgresql://postgres:your-db-password@db.your-project-ref.supabase.co:5432/postgres

# Optional: Google OAuth for local development.
# Leave blank if you will log in with email/password only.
SUPABASE_AUTH_EXTERNAL_GOOGLE_CLIENT_ID=
SUPABASE_AUTH_EXTERNAL_GOOGLE_SECRET=
```

Important:

- `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, and `SUPABASE_SERVER_URL` are local values generated by the project.
- Do not paste production values into those four variables for local development.
- `SUPABASE_PRODUCTION_DB_URL` is only used by `npm run db:sync:prod-to-local`.

## How To Run The Project Locally

If you want the shortest path, run exactly this:

```bash
npm install
cp .env.local.example .env.local
npm run local:up
```

What `npm run local:up` does:

1. starts Supabase local in Docker
2. writes the local Supabase credentials into `.env.local`
3. regenerates `src/types/supabase.ts`
4. builds the app container
5. starts the app container

When it finishes:

- the app should be at `http://localhost:3000`
- the login page should be at `http://localhost:3000/login`
- the local Supabase API should be at `http://127.0.0.1:54321`
- the local Postgres database should be at `127.0.0.1:54322`

If you want to run the same steps manually:

```bash
npm run local:setup
docker compose -f compose.local.yml up -d --build app
```

To stop everything:

```bash
npm run local:down
```

If you change [supabase/config.toml](/Users/fitodac/.codex/worktrees/4e51/skintrack/supabase/config.toml), restart the local stack so Supabase Auth picks up the new config:

```bash
npm run local:down
npm run local:up
```

Useful local commands:

```bash
npm run db:status
npm run db:reset
npm run db:stop
docker compose -f compose.local.yml ps
docker compose -f compose.local.yml logs -f app
```

## How To Clone Production Data Into Local

If you want your local database to contain production data, run:

```bash
npm run db:sync:prod-to-local
```

This command does all of this:

1. starts local Supabase if it is not already running
2. resets the local database from the repo migrations
3. reads the remote `public` tables
4. dumps production data
5. restores that dump into your local Supabase database

What it copies:

- all `public.*` data
- `auth.users`
- `auth.identities`

What it does not copy:

- active sessions
- refresh tokens

Requirements:

- `SUPABASE_PRODUCTION_DB_URL` must exist in `.env.local` or your shell
- `pg_dump` must be installed
- `psql` must be installed

Very important:

- use the Supabase direct connection string
- if the host is `db.<project-ref>.supabase.co`, the username must be `postgres`
- do not use a pooler username like `postgres.<project-ref>` with the direct host

Correct example:

```env
SUPABASE_PRODUCTION_DB_URL=postgresql://postgres:your-db-password@db.your-project-ref.supabase.co:5432/postgres
```

If production has schema drift compared to [20260312180000_init_skintrack.sql](/Users/fitodac/.codex/worktrees/4e51/skintrack/supabase/migrations/20260312180000_init_skintrack.sql), fix that first. The sync command copies data. It does not repair schema drift.

## How To Reset Local Database

If you want to wipe the local database and rebuild it from migrations only, run:

```bash
npm run db:reset
```

Use this when:

- you want a clean local schema
- you do not want production data loaded
- you want to test migrations from scratch

## How To Connect With TablePlus

You can inspect the local Supabase database with TablePlus or any PostgreSQL client.

Use these values:

- Host: `127.0.0.1`
- Port: `54322`
- User: `postgres`
- Password: `postgres`
- Database: `postgres`
- SSL: disabled

Full connection string:

```txt
postgresql://postgres:postgres@127.0.0.1:54322/postgres
```

This is the local database created by Supabase CLI.

Recommended flow:

1. run `npm run local:up`
2. if needed, run `npm run db:sync:prod-to-local`
3. open TablePlus
4. create a new PostgreSQL connection
5. paste the values above

## Authentication Notes

- public signup stays disabled
- users are invited by a superadmin
- email/password login works for invited users
- Google OAuth is optional in local, but required if you want full parity with production OAuth flows

In local Supabase config:

- project-wide signup stays disabled
- email/password auth stays enabled so invited users can log in

For local Google OAuth, your Google client must allow:

- `http://127.0.0.1:54321/auth/v1/callback`
- `http://localhost:3000/auth/callback`

On first successful login, the onboarding flow creates or syncs the profile row through the database logic already defined in the project.

## Deploy To Vercel

This project is meant to deploy on Vercel.

### 1. Create the environment variables in Vercel

Add these values in the Vercel project settings:

```env
NEXT_PUBLIC_APP_URL=https://your-production-domain.com
NEXT_PUBLIC_SUPABASE_URL=https://your-project-ref.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=your-production-anon-key
SUPABASE_SERVICE_ROLE_KEY=your-production-service-role-key
```

If production uses Google login, also add:

```env
SUPABASE_AUTH_EXTERNAL_GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
SUPABASE_AUTH_EXTERNAL_GOOGLE_SECRET=your-google-client-secret
```

### 2. Configure redirect URLs

Production should allow:

- `https://your-production-domain.com/auth/callback`

Configure that URL in:

- Supabase Auth
- your Google OAuth app, if Google login is enabled

### 3. Deploy

Push your branch to GitHub and let Vercel build it, or connect the repo directly in Vercel.

## Notes

- [src/middleware.ts](/Users/fitodac/.codex/worktrees/4e51/skintrack/src/middleware.ts) is required for session refresh and protected routing.
- [src/types/supabase.ts](/Users/fitodac/.codex/worktrees/4e51/skintrack/src/types/supabase.ts) is regenerated from the local Supabase stack with `npm run db:types`.
- [supabase/config.toml](/Users/fitodac/.codex/worktrees/4e51/skintrack/supabase/config.toml) contains the local Supabase CLI configuration.
- `SUPABASE_SERVER_URL` exists only so the app container can reach Supabase on the host machine.
- The project does use containers. What it does not use is a fully self-hosted custom Supabase Docker stack, because Supabase CLI already provides the local Supabase containers with much less setup complexity.

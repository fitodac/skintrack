# Setup

## Prerequisites

Install these tools before the first boot:

- Node.js 20+
- npm
- Docker Desktop
- PostgreSQL client tools: `psql` and `pg_dump`

The project uses Docker in two places:

- Supabase local runs in Docker containers managed by the Supabase CLI
- the Next.js app runs in the local `skintrack-app` container

## Install dependencies

```bash
npm install
```

## Create local environment variables

```bash
cp .env.local.example .env.local
```

Set only the project-specific values before the first run:

```bash
NEXT_PUBLIC_APP_URL=http://localhost:3000
SUPABASE_PRODUCTION_DB_URL=postgresql://postgres:your-db-password@db.your-project-ref.supabase.co:5432/postgres
SUPABASE_AUTH_EXTERNAL_GOOGLE_CLIENT_ID=your-google-client-id.apps.googleusercontent.com
SUPABASE_AUTH_EXTERNAL_GOOGLE_SECRET=your-google-client-secret
```

These values are generated or overwritten by local setup and should not be filled with production credentials:

```bash
NEXT_PUBLIC_SUPABASE_URL
NEXT_PUBLIC_SUPABASE_ANON_KEY
SUPABASE_SERVICE_ROLE_KEY
SUPABASE_SERVER_URL
```

## Fastest local boot

```bash
npm run local:up
```

This single command does all of this:

- starts local Supabase in Docker
- writes local Supabase credentials into `.env.local`
- regenerates `src/types/supabase.ts`
- builds the app container
- starts the app container

After it finishes:

- app: `http://localhost:3000`
- login: `http://localhost:3000/login`
- local Supabase API: `http://127.0.0.1:54321`
- local Postgres: `127.0.0.1:54322`

## Manual path

If you want the same flow with explicit steps:

```bash
npm run local:setup
docker compose -f compose.local.yml up -d --build app
```

`npm run local:setup` expands to:

```bash
npm run db:start
npm run db:env:local
npm run db:types
```

## Local env behavior

`npm run db:env:local` reads `supabase status -o env` and updates these keys in `.env.local`:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_SERVER_URL`

`SUPABASE_SERVER_URL` is important for the Dockerized app. It rewrites `127.0.0.1` or `localhost` to `host.docker.internal` so the `skintrack-app` container can still reach the Supabase API.

## Local schema and auth config

The local Supabase setup is defined in:

- `supabase/config.toml`
- `supabase/migrations/20260312180000_init_skintrack.sql`

The current config keeps local auth close to production:

- project-wide signup disabled at `[auth]`
- email auth enabled so invited users can accept invites
- Google OAuth enabled through environment variables
- redirect callback at `http://127.0.0.1:54321/auth/v1/callback`
- app callback allowed at `/auth/callback`

If you change `supabase/config.toml`, restart the local stack so auth picks up the new config:

```bash
npm run local:down
npm run local:up
```

## Sync production data into local Supabase

To fully replace local data with production data:

```bash
npm run db:sync:prod-to-local
```

This command:

1. starts local Supabase if it is not running
2. resets the local database from migrations
3. dumps all `public` tables from production
4. also copies `auth.users` and `auth.identities`
5. restores the dump into local Supabase

Requirements:

- `SUPABASE_PRODUCTION_DB_URL` must be present
- `pg_dump` and `psql` must be installed locally

If the host is `db.<project-ref>.supabase.co`, the username must be `postgres`. Do not use a pooler-style username with the direct host.

The sync intentionally does not copy active sessions or refresh tokens.

The command copies data only. If production drifted from the repo migrations, fix schema drift first.

## Local DB clients

To inspect local data with TablePlus or any PostgreSQL client:

```bash
Host=127.0.0.1
Port=54322
User=postgres
Password=postgres
Database=postgres
SSL=disabled
```

Connection URL:

```txt
postgresql://postgres:postgres@127.0.0.1:54322/postgres
```
